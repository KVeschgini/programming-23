## Die letzte Programmierübung

Die letzte Programmierübung in diesem Kurs besteht darin, ein kleines eigenes Spiel mit Pygame zu erstellen. Die Übungsvorlage enthält einige Bilder, die Sie verwenden können. Bitte verwenden Sie keine anderen Bilder in Ihrem Spiel, da andere Ihr Spiel sonst nicht allein auf der Grundlage des Quellcodes testen können.

Das Thema des Spiels bleibt Ihnen überlassen, aber es sollte in etwa so komplex sein wie das Sokoban-Beispiel in diesem Teil des Materials. Das Spiel sollte die folgenden Merkmale aufweisen:

* Das Spiel hat ein Sprite, das der Spieler auf irgendeine Weise bewegen kann
* Das Spiel hat einige sammelbare Gegenstände und/oder Gegner
* Dem Spieler muss eine klare Aufgabe im Spiel gestellt werden
* Das Spiel enthält einen Zähler, der dem Spieler mitteilt, wie er im Spiel abschneidet
* Der Quellcode für das Spiel ist wie im Sokoban-Beispiel in Funktionen unterteilt

Wenn das Spiel fertig ist, reichen Sie es mit dem folgenden Formular zur Peer-Review ein. Machen Sie sich danach mit den Spielen von zwei Ihrer Kurskollegen vertraut, geben Sie den Spielen eine Note, die Sie für angemessen halten, und schreiben Sie eine kurze Rezension des Spiels. Bitte berücksichtigen Sie bei der Benotung und Rezension der Spiele sowohl, wie interessant und spielbar das Spiel war, als auch, wie lesbar der Programmcode ist.

### Einreichen des Spiels

Sie sollten den Quellcode Ihres Spiels über die TMC-Paste-Funktion einreichen und den Link, den Sie erhalten, in das untenstehende Formular einfügen.

Klicken Sie bei geöffneter Lösung im Editor auf die TMC-Menüschaltfläche in Visual Studio Code (neben dem Augensymbol). Dies öffnet ein Menü, in dem Sie nach der TMC-Paste-Funktion suchen können, indem Sie `send` eingeben:

## Wie Teil 14 benotet wird

Das Ziel dieses Teils ist es, ein kleines, funktionierendes Spiel zu bauen. Da dieser Teil nur eine einzige Aufgabe enthält, *erhalten Sie Punkte für den Aufwand*. Bitte reichen Sie Ihr Spiel auch dann ein, wenn Sie es nicht ganz so zum Laufen bringen konnten, wie Sie es wollten.

Wenn Ihr Spiel nicht so funktioniert, wie Sie es beabsichtigt haben, fügen Sie bitte Kommentare in Ihren Code ein, wie es funktionieren _sollte_ oder welche Funktionen Ihrer Meinung nach fehlen, die Sie aber nicht implementieren konnten.

Wenn ein Spiel, das Sie rezensieren, unvollständig ist oder nicht ganz richtig funktioniert, fügen Sie bitte Kommentare darüber hinzu, was Ihrer Meinung nach hätte getan werden können, um es zum Laufen zu bringen, falls dies überhaupt möglich ist.

## Einige Spielideen

Es bleibt ganz Ihnen überlassen, worum es in Ihrem Spiel geht, aber hier sind einige Ideen, die Ihnen den Einstieg erleichtern sollen. Denken Sie an das Material in Teil 13; diese Beispiele können Ihnen auch bei der Ausarbeitung Ihrer Ideen helfen.

### Ein Sammelspiel

* Der Spieler bewegt den Roboter mit den Pfeiltasten.
* Eine Münze erscheint an einer zufälligen Stelle auf dem Bildschirm. Wenn der Roboter sie erreicht, bewegt sich die Münze an eine neue Stelle.
* Es befinden sich auch Monster auf dem Bildschirm, denen der Roboter ausweichen muss.

### Ein Münzregen

* Der Spieler bewegt den Roboter am unteren Bildschirmrand nach links und rechts.
* Münzen regnen vom Himmel. Der Roboter muss diese einsammeln.
* Auch Monster regnen vom Himmel. Der Roboter muss diesen ausweichen.

## Peer-Review

Sie sollten das Spiel basierend auf den folgenden Kriterien bewerten:

* Wie sieht das Spiel aus?
* Ist das Spiel *spielbar*? Macht es Spaß zu spielen und ist es einfach zu bedienen?
* Ist die Spielidee interessant?
* Wie gut ist es programmiert? Ist der Code lesbar und macht er angemessenen Gebrauch von Klassen und Funktionen?

Eine gute Rezension weist in der Regel sowohl auf gute Merkmale als auch auf einige Verbesserungsvorschläge hin.
